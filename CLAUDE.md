## Контекст проекта

Веб-тренажёр CW. Форк [sc0tfree/morsewalker](https://github.com/sc0tfree/morsewalker),
развивается R9OGL. Прод: https://morse.r9o.ru

**Позиционирование (важно для решений):** это НЕ клон Morse Runner для
опытных контестменов. Цель — удобный тренажёр для **начинающих**, с обучающими
методиками. Но там, где речь о стандартах живого эфира (раскладка функциональных
клавиш, формат обменов, правила подсчёта), нужно следовать общепринятому —
чтобы новичок сразу привыкал к правильному, а не переучивался потом.

Стек: ванильный ES6+, Webpack 5, Bootstrap 5 (bootswatch/cerulean), Web Audio API.
Тестов нет. Линтера нет, только Prettier + husky pre-commit.

## Порядок работы

```bash
npm install
npm start          # dev-сервер
npm run build      # прод-сборка в dist/
npm run format     # prettier
```

Перед коммитом: `npm run format`. Проверять изменения вручную в браузере —
автотестов нет, консоль браузера сейчас основной инструмент отладки.

Комментарии и логи в коде — на русском, с эмодзи-маркерами (`✅`, `🆕`, `🏆`).
Держаться этого стиля, чтобы диффы читались единообразно.

## 🔴 Критично: сломанный код в app.js (ИСПРАВЛЕНО)

Тело функции `applyModeSettings()` продублировано на верхнем уровне модуля —
после закрывающей скобки функции идёт копия её содержимого:

```js
}                                  // конец applyModeSettings

// 🆕 Если режим RDA — включить Russian Only

  ...
}
```

Это ровно та ошибка `ERROR: mode is not defined`, которая записана в

объявлено ниже точки падения на верхнем уровне, не инициализируется.

регистрируется выше по файлу, а объявления функций поднимаются (hoisting).

это точная копия того, что уже есть внутри функции.

Второй артефакт того же копипаста: `updateScoreboard()` имеет два одинаковых

## Что чинить дальше (по приоритету)

### 1. Горячие клавиши — привести к стандарту контест-логгеров

`src/js/hotkeys.js`. Текущая раскладка расходится с общепринятой, и это
главная претензия из обратной связи от опытных контестменов (R8OA).

Стандарт, единый для Morse Runner, N1MM, DXLog и живого эфира:

| Клавиша | Значение |
|---|---|
| F1 | CQ (общий вызов) |
| F2 | твой обмен (599 + номер / имя+регион по режиму) |
| F3 | TU / завершение QSO |
| F4 | твой позывной |
| F5 | позывной корреспондента |
| F6–F12 | свободно программируются |
| Enter | ESM — отправить сообщение, зависящее от стадии QSO |
| Esc | стоп передачи |
| Insert | позывной корреспондента + обмен одной посылкой |
| `+` / `-` | скорость |

Ключевые замечания:
- **`QRS` на F3 — ошибка.** В контестах QRS не запрашивают. F3 = TU.
- **`AGN?` на F2 — ошибка.** F2 — это твой обмен: то, чем ты отвечаешь,
  когда корреспондент просит повторить номер. Запрос повтора — это `?` или
  `AGN`, вешать на F7/F8.
- QRS/AGN уместны в обучающем режиме для новичков, но не в контестном.

Предлагаемое решение — **две раскладки**, переключаемые вместе с режимом:
Contest Mode (стандарт выше) и Training Mode (мягкая, с AGN?/QRS для новичков).
Раскладку выводить в UI и в Help.

Технические баги того же файла:
- `HotkeyManager` получает объект DOM-элементов в конструктор, но игнорирует
  его и всё равно ходит через `document.getElementById`. Либо использовать,
  либо не передавать.
- **Enter обрабатывается дважды**: глобально в `hotkeys.js` и локально на
  `responseField` в `app.js` → двойной вызов `send()`.
- **Ctrl+R перехватывает перезагрузку страницы.** Взять другую комбинацию
  (в логгерах для очистки полей — F12 / Ctrl+W «wipe»).
- Хоткеи срабатывают при наборе в любом поле формы, включая настройки станции.
  Нужна проверка `e.target` / фокуса.
- Нет проверки `e.repeat` — удержание клавиши шлёт поток кликов.

### 2. ESM (Enter Sends Message)

Самая ценная фича для обучения контестингу и то, чего сейчас нет
принципиально. В логгерах Enter отправляет разное в зависимости от стадии QSO:
поле позывного пустое → CQ; введён позывной → его позывной + твой обмен;
обмен принят → TU и следующий QSO. Сейчас Enter всегда просто «Send».

Реализовать как конечный автомат стадий QSO в `app.js`, а не набором `if`.
Состояние QSO уже частично есть (`readyForTU`, `activeStationIndex`) — его
надо формализовать: `IDLE → CQ_SENT → CALL_COPIED → EXCHANGE_SENT → READY_FOR_TU`.

### 3. Подсчёт очков — расхождения с реальностью

`src/js/scoring.js`:

- **`mistakes` никогда не увеличивается.** `recordMistake()` не вызывается
  ниоткуда → «Точность» всегда 100%. Нужно вызывать при неверно принятом
  позывном/обмене (данные для этого уже считаются в
  `app.js::compareExtraInfo()`, но результат сверки только красится в таблице
  и никуда не идёт).
- **Множители не считаются для `contest`, `sst`, `hst`** → `totalScore =
  points × 1`. Для этих режимов либо ввести множители, либо честно показывать
  «множители не применяются».
- **Дубль засчитывается в лог.** `addQSO()` возвращается рано при дубле, но
  `tu()` до этого уже увеличил `totalContacts` и добавил строку в таблицу.
  Строку дубля надо помечать («DUPE, 0 очков»), а не молча расходиться со
  счётчиком.
- **`extractPrefix()` — жадный regex** `/^[A-Z0-9]+\d/`. На `RA9ABC` вернёт
  `RA9` (верно), но на позывных с цифрой в суффиксе или дробных (`R9O/3`,
  `UA0ABC/9`) даст мусор. Правила WPX для дробных позывных отдельные — надо
  реализовать явно. Дубликат этой же функции лежит в `modes.js` — оставить одну.
- `case 'wpx':` объявляет `const prefix` без обёртки в блок `{}` внутри
  `switch` — работает, но это ловушка TDZ. Обернуть.
- **Очки в одиночных режимах (`single`, `hst`) не начисляются вообще** —
  ветка `send()` без `showTuStep` не вызывает `addQSO()`.

### 4. HST-режим — сейчас это заглушка

`modes.hst` передаёт только позывные, `showTuStep: false`, счёта нет.
Настоящий HST — дисциплина на скорость приёма с жёстким регламентом.
Что нужно: таймер сессии, счёт по количеству правильно принятых позывных за
время, растущая скорость, отсутствие подсказок. Без этого называть режим HST
некорректно — лучше временно переименовать в «Скоростной приём» или пометить
как WIP.

### 5. WPX-режим

- Серийный номер станции генерируется случайно в диапазоне 01–30
  (`stationGenerator.js`) и не инкрементируется. В WPX номера идут
  последовательно от 001 — новичок привыкнет к неправильному.
- Нет проверки на повтор позывного внутри пачки → возможны дубли
  в одном pile-up.
- Очки WPX в реальности зависят от диапазона и континента корреспондента —
  сейчас всегда 1. Либо упростить сознательно и написать об этом в Help,
  либо реализовать.

### 6. Локализация

- Названия режимов **HST и WPX не переводятся**: в
  `localization/index.js::updateUI()` вызовы `updateLabel()` есть только для
  шести старых режимов. Добавить `modeHst` / `modeWpx`.
- `updateUI()` наполовину построена на поиске элементов по подстроке текста
  (`label.textContent.includes('QRN')`). Хрупко: сломается при любой правке
  вёрстки и уже сейчас работает только в одну сторону. Перевести всё на
  `data-i18n`.
- `updateHelpModal()` адресует элементы по индексам (`helpSteps[1]`,
  `helpTexts[3]`) — сломается при добавлении абзаца в Help. Тоже на `data-i18n`.

## Соглашения

- **Новый режим** добавляется в `modes.js` и нигде больше не хардкодится.
  Если для режима понадобилась ветка `if (currentMode === '...')` в `app.js` —
  это сигнал, что конфиг режима надо расширить. Сейчас такие ветки уже есть
  (`sst`/`pota`/`rda` в `tu()`, `rda` в `applyModeSettings()`) — их стоит
  вынести в конфиг.
- Строки, видимые пользователю, — только через `i18n.t()` или `data-i18n`.
  Не хардкодить текст в JS.
- Тайминги аудио строятся на возвращаемом времени `playSentence()`,
  не на `setTimeout`.
- Любое изменение игрового цикла проверять во всех восьми режимах —
  режимы делят один код.
