## Контекст проекта

Веб-тренажёр CW. Форк [sc0tfree/morsewalker](https://github.com/sc0tfree/morsewalker),
развивается R9OGL. Прод: https://morse.r9o.ru

**Позиционирование (важно для решений):** это НЕ клон Morse Runner для
опытных контестменов. Цель — удобный тренажёр для **начинающих**, с обучающими
методиками. Но там, где речь о стандартах живого эфира (раскладка функциональных
клавиш, формат обменов, правила подсчёта), нужно следовать общепринятому —
чтобы новичок сразу привыкал к правильному, а не переучивался потом.

Стек: ванильный ES6+, Webpack 5, Bootstrap 5 (bootswatch/cerulean), Web Audio API.
Тесты — только на чистых функциях (`npm test`, встроенный раннер Node).
Линтера нет, только Prettier + husky pre-commit.

## Порядок работы

```bash
npm install
npm start          # dev-сервер
npm run build      # прод-сборка в dist/
npm run format     # prettier
npm test           # тесты scoring.js (node --test)
```

Перед коммитом: `npm run format` и `npm test`. Игровой цикл автотестами не
покрыт — его по-прежнему проверять руками в браузере, консоль остаётся
основным инструментом отладки. Логика, которую можно вынести в чистую
функцию, должна ехать в тесты, а не в консоль.

Комментарии и логи в коде — на русском, с эмодзи-маркерами (`✅`, `🆕`, `🏆`).
Держаться этого стиля, чтобы диффы читались единообразно.

## 🔴 Критично: сломанный код в app.js (ИСПРАВЛЕНО)

> Оставлено как описание симптома — если ошибка всплывёт снова, искать здесь.

**`src/js/app.js`, примерно строки 366–415.**

Тело функции `applyModeSettings()` продублировано на верхнем уровне модуля —
после закрывающей скобки функции идёт копия её содержимого:

```js
}                                  // конец applyModeSettings

// 🆕 Если режим RDA — включить Russian Only
if (mode === 'rda') {              // ← ReferenceError: mode is not defined
  ...
}
const tuButton = document.getElementById('tuButton');
...
modeResultsHeader.textContent = config.resultsHeader;   // ← config тоже не определён
```

Это ровно та ошибка `ERROR: mode is not defined`, которая записана в
`Текстовый документ.txt`. Модуль падает при вычислении, поэтому всё, что
объявлено ниже точки падения на верхнем уровне, не инициализируется.
Приложение частично работает только потому, что слушатель `DOMContentLoaded`
регистрируется выше по файлу, а объявления функций поднимаются (hoisting).

**Починка: удалить блок целиком.** Никакой уникальной логики в нём нет —
это точная копия того, что уже есть внутри функции.

Второй артефакт того же копипаста: `updateScoreboard()` имеет два одинаковых
JSDoc-блока подряд.

## Что чинить дальше (по приоритету)

### 1–2. Хоткеи и ESM — СДЕЛАНО

Раскладка приведена к стандарту (F1 CQ, F2 обмен, F3 TU, F4 свой позывной,
F5 его позывной, Insert = F5+F2), Enter работает как ESM. Замечания R8OA
учтены: QRS убран из контестной раскладки, F2 больше не AGN.

Что осталось спорным и требует решения:
- **F6, F9–F11 не заняты.** В логгерах они программируются пользователем.
  Стоит ли давать редактор макросов новичку — вопрос открытый.
- **Раскладка не привязана к режиму**, переключается вручную тумблером.
  Возможно, `single` должен включать обучающую автоматически.

### 3. Подсчёт очков — ИСПРАВЛЕНО, осталась проверка на живых данных

Всё из прежнего списка починено и покрыто тестами (`npm test`). Что стоит
перепроверить руками в браузере:

- Точность и «Ошибки» в Scoreboard после заведомо неверно принятого номера.
- Строка дубля: помечается бейджем `DUPE`, жёлтая, очков не даёт.
- Множители в `contest`/`sst` показывают прочерк, а не 0 или 1.

Спорное решение, требующее мнения контестмена: **неверно принятый обмен
полностью не засчитывается** (как в Morse Runner), а не «засчитывается со
штрафом». Для новичка это может быть слишком жёстко — возможно, стоит
сделать поведение настраиваемым или мягким в обучающем режиме.

### 4–5. HST и WPX — ЗАКРЫТЫ до готовности

Радиокнопки `disabled`, бейдж «СКОРО», в `app.js` список `DISABLED_MODES`
не даёт восстановить режим из localStorage. Конфиги в `modes.js` и правила
в `MODE_RULES` оставлены — закрыт только вход.

Почему закрыты:

- **HST.** Сейчас это заглушка: передаются только позывные, `showTuStep: false`,
  счёта нет. Настоящий HST — дисциплина на скорость приёма с жёстким
  регламентом (чемпионаты России, Европы, мира). Нужны таймер сессии, счёт по
  количеству правильно принятых позывных за время, растущая скорость,
  отсутствие подсказок. **Перед реализацией — узнать регламент у R8OA.**
- **WPX.** Серийный номер станции генерируется случайно 01–30
  (`stationGenerator.js`) и не инкрементируется, хотя в WPX номера идут
  последовательно от 001. Нет проверки на повтор позывного внутри пачки.
  Очки в реальности зависят от диапазона и континента корреспондента —
  сейчас всегда 1. Ориентир по правилам — Кубок RU_CW по WPX (закреплённое
  сообщение в теме MorseRunner/RufzXP).

### 5а. Аккаунты и таблица лидеров (бэклог, крупное)

Идея: соревновательный режим, ради которого всё и затевается — общая таблица
лидеров, возможно проведение соревнований по этому тренажёру.

Что это меняет архитектурно (сейчас проект — статика без бэкенда):

- Нужен **бэкенд**: хранение результатов, аутентификация. Появляется сервер,
  БД, миграции — то, чего в проекте пока нет вообще.
- **Идентификация по позывному.** Позывной — не секрет, любой может ввести
  чужой. Для таблицы лидеров нужно подтверждение владения: почта из QRZ,
  либо ручная модерация, либо OAuth через сторонний сервис.
- **Античит — главная проблема.** Клиент знает правильный ответ (позывные и
  обмены генерируются в браузере), поэтому счёт с клиента доверять нельзя
  в принципе. Варианты: генерация сессии на сервере с проверкой ответов,
  либо честное разделение — «локальные рекорды» без модерации и отдельный
  «официальный зачёт» с серверными сессиями.
- **Уже есть телеметрия** на `stats.<hostname>/api/submit` (шлёт позывной и
  режим). Это зародыш бэкенда, но и повод сначала решить вопрос с приватностью:
  сейчас отправка идёт без уведомления пользователя.

Порядок разумной реализации: сначала локальные рекорды в `localStorage`
(ноль инфраструктуры, сразу даёт мотивацию), потом серверные сессии для
официального зачёта. Не начинать с аккаунтов.

### 6. Локализация

- Названия режимов **HST и WPX не переводятся**: в
  `localization/index.js::updateUI()` вызовы `updateLabel()` есть только для
  шести старых режимов. Добавить `modeHst` / `modeWpx`.
- `updateUI()` наполовину построена на поиске элементов по подстроке текста
  (`label.textContent.includes('QRN')`). Хрупко: сломается при любой правке
  вёрстки и уже сейчас работает только в одну сторону. Перевести всё на
  `data-i18n`.
- `updateHelpModal()` адресует элементы по индексам (`helpSteps[1]`,
  `helpTexts[3]`) — сломается при добавлении абзаца в Help. Тоже на `data-i18n`.

### 8. Прочее

- Нет `nonRussianSerialNumber` сброса в `resetGameState()` (есть только в
  `reset()`) — счётчик течёт между сессиями.
- Телеметрия на `stats.<hostname>/api/submit` отправляет позывной без
  уведомления пользователя. Упомянуть в README/Help или сделать opt-in.
- Нет автотестов. Первое, что стоит покрыть, — `scoring.js` и

  (vitest/jest).
- Нет ESLint. Он бы поймал `mode is not defined` до коммита.

## Соглашения

- **Новый режим** добавляется в `modes.js` и нигде больше не хардкодится.
  Если для режима понадобилась ветка `if (currentMode === '...')` в `app.js` —
  это сигнал, что конфиг режима надо расширить. Сейчас такие ветки уже есть
  (`sst`/`pota`/`rda` в `tu()`, `rda` в `applyModeSettings()`) — их стоит
  вынести в конфиг.
- Строки, видимые пользователю, — только через `i18n.t()` или `data-i18n`.
  Не хардкодить текст в JS.
- Тайминги аудио строятся на возвращаемом времени `playSentence()`,
  не на `setTimeout`.
- Любое изменение игрового цикла проверять во всех восьми режимах —
  режимы делят один код.
